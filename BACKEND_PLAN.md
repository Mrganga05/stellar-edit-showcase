# Raqvine Backend Implementation Plan

## Architecture

- Runtime: TanStack Start server routes
- API style: Same-origin JSON REST endpoints under `/api`
- Database: SQLite for local and single-instance deployment
- Validation: Zod at every write boundary
- IDs: UUID strings generated by the server
- Timestamps: UTC ISO-8601 strings
- Admin protection: `x-admin-key` header for non-public operations
- Client data: TanStack Query with loading, error, and fallback states

## Frontend Feature Analysis

### Portfolio Projects

- Frontend usage: Project grid, hover preview, and project detail modal
- Endpoints:
  - `GET /api/portfolio`
  - `POST /api/portfolio`
  - `GET /api/portfolio/:id`
  - `PUT /api/portfolio/:id`
  - `DELETE /api/portfolio/:id`
- Tables: `portfolio_projects`, optional relationship to `services`
- Validation:
  - Required title, category, thumbnail URL/path, video URL, description
  - URL/path validation for media
  - Maximum field lengths and valid boolean/sort values
- Business logic:
  - Public reads return published projects in display order
  - Admin writes require authentication
  - Deletes return `404` when the project does not exist

### Before/After Projects

- Frontend usage: Interactive comparison slider with two synchronized videos
- Endpoints:
  - `GET /api/comparisons`
  - `POST /api/comparisons`
  - `GET /api/comparisons/:id`
  - `PUT /api/comparisons/:id`
  - `DELETE /api/comparisons/:id`
- Table: `before_after_projects`
- Validation:
  - Required project title, before video, after video, and description
  - Before and after media URLs must be valid
- Business logic:
  - Public reads only include published comparisons
  - Results are ordered by `sort_order`

### Testimonials

- Frontend usage: Client testimonial carousel
- Endpoints:
  - `GET /api/testimonials`
  - `POST /api/testimonials`
  - `GET /api/testimonials/:id`
  - `PUT /api/testimonials/:id`
  - `DELETE /api/testimonials/:id`
- Table: `testimonials`
- Validation:
  - Required client name, company, review, profile image
  - Rating must be an integer from 1 through 5
- Business logic:
  - Public reads only include published testimonials
  - Admin controls order and publication state

### Contact Form

- Frontend usage: Project inquiry form
- Endpoints:
  - `POST /api/contact`
  - `GET /api/contact`
  - `GET /api/contact/:id`
  - `PUT /api/contact/:id`
  - `DELETE /api/contact/:id`
- Table: `contact_inquiries`
- Validation:
  - Required name, email, project type, budget, timeline, and details
  - Normalized lowercase email
  - Enumerated project type, budget, timeline, and status values
- Business logic:
  - Public can create inquiries
  - Admin can list, inspect, update status/notes, and delete
  - New inquiries default to `new`

### Sample Edit Requests

- Frontend usage: Free sample request CTA and request form
- Endpoints:
  - `POST /api/sample-requests`
  - `GET /api/sample-requests`
  - `GET /api/sample-requests/:id`
  - `PATCH /api/sample-requests/:id`
  - `DELETE /api/sample-requests/:id`
- Table: `sample_edit_requests`
- Validation:
  - Required name, email, footage link, and message
  - Footage link must be an HTTP(S) URL
  - Status must match the request workflow
- Business logic:
  - Public can submit requests
  - Admin can retrieve requests and update status/notes
  - New requests default to `new`

### Services

- Frontend usage: Service cards and portfolio/contact categorization
- Endpoints:
  - `GET /api/services`
  - `POST /api/services`
  - `GET /api/services/:id`
  - `PUT /api/services/:id`
  - `DELETE /api/services/:id`
- Table: `services`
- Validation:
  - Unique slug, required title/description/icon key
- Business logic:
  - Public reads include active services only
  - A deleted service sets linked project/contact service IDs to null

### FAQ

- Frontend usage: Expandable FAQ accordion
- Endpoints:
  - `GET /api/faqs`
  - `POST /api/faqs`
  - `GET /api/faqs/:id`
  - `PUT /api/faqs/:id`
  - `DELETE /api/faqs/:id`
- Table: `faqs`
- Validation:
  - Required question and answer with bounded lengths
- Business logic:
  - Public reads include published entries in display order

### Admin Dashboard

- Frontend usage: Operational summary for a future admin UI
- Endpoint:
  - `GET /api/dashboard`
- Tables: Aggregates `portfolio_projects`, `testimonials`, `contact_inquiries`, and `sample_edit_requests`
- Validation: Admin API key
- Business logic:
  - Returns all four counts in one database round trip

## API Response Contract

Successful response:

```json
{
  "success": true,
  "data": {}
}
```

Error response:

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": {}
  }
}
```

## Security And Operations

- Require `ADMIN_API_KEY` in production.
- Never expose the admin key through `VITE_*` variables.
- Validate all request bodies and route IDs.
- Use prepared SQL statements only.
- Limit request body size at the reverse proxy in production.
- Add rate limiting and bot protection to public form endpoints before launch.
- Use an external managed database when deploying more than one server instance.
- Add object storage and signed uploads before accepting direct media files.
